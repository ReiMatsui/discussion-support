## 2. 重要な抜け漏れ (優先度高 → 低)

### A. ファシリテーションエージェントが研究計画書通りに作られていない (優先度: 高)

研究計画書 §4.1 の最後で、ファシリテーションエージェントは:
1. 統合グラフ全体を読みながら、いつ何を参加者に提示するかを判断する**中央調停者**
2. 参加者の最新発言を支持・攻撃するノードを取り出して提示
3. 議論が一方の立場に偏っている状態を**構造的に検知して反対側を補強**
4. 議論ステージ (発散・収束) に応じて**介入方針を切り替え**

と定義されている。現状実装されているのは **2 だけ** (隣接エッジの取り出し)。3 と 4 は未着手。

これは論文の提案手法の核の 1 つ。RQ3 (介入の透明性) も、ファシリテーションが「ただ近傍を出すだけ」では成立しない。**提案手法の完成度を上げる上で最優先**。

### B. LLM-as-judge による主観指標代理計測 (優先度: 高)

研究計画書 §5.1 の主観指標 (議論満足度・情報有用性・反対意見理解度・立場自信度・介入透明性) を、LLM-as-judge で代理計測する仕組みが未。これがないと **RQ1〜RQ3 のシミュレーション段階での仮説検証ができない**。

タスク化済み (M2.5)。多数回ラン (M2.6) と組み合わせて、3 条件比較に統計的な裏付けが付く。

### C. 段階A (技術検証) が未実施 (優先度: 高)

研究計画書 §5.2 段階A:
- 統合議論グラフ構築精度を **人手アノテーションとの一致率**で評価
- claim/premise の F1、連結エッジの precision/recall、リアルタイム遅延

これがないと、下流の評価 (B/C) の前提が崩れる。「LinkingAgent が実用的な精度で動いている」という主張に裏付けがない。少なくとも 30〜50 ペアでの精度評価は必要。

### D. L3 自然言語要約と L4 振り返りビュー (優先度: 中)

§4.4 の 4 層の上半分 (L1, L2) しか実装されていない。

- **L3**: 議論区切りで「論点 A vs B、支持 3 件 / 2 件、未応答反論 1 件」のような自然文を全員に提示。これは比較的小さく、`graph_metrics` を文字列化するだけで原型ができる
- **L4**: 議論後に参加者ごとに「あなたが反応しなかった論点」「立場と対立した主張」を提示

両方とも提案手法の重要な構成要素。論文に書く以上、最低限のプロトタイプは必要。

### E. 介入ログの永続化 (優先度: 中)

C3 (介入の透明性) の主張を担保するには、いつ誰に何が提示されたかを永続化する必要がある。現状は Live UI で表示して消える。

`data/runs/<run_id>/interventions.jsonl` のようなログを取り、後から再現できるようにすべき。

### F. 多数回ラン executor (優先度: 中)

LLM の確率的ゆらぎが大きいので、3 条件を 1 回ずつ比較しても何も主張できない。`N=10` 程度で平均と分散を取る仕組み (M2.6) が要る。これも CLI / Streamlit から叩けると論文用の数字が一気に出せる。

### G. 政策トピックの文書 (優先度: 中)

`policy_ai_lecture_personas` (生成 AI 大学講義利用) のペルソナは用意したが、対応する文書 (`data/docs/` 配下) はカフェテリア向けしかない。FlatRAG / FullProposal を政策トピックで動かすには 3〜4 本の関連文書が要る。

### H. Web 検索エージェント (優先度: 中)

§4.1 の 5 エージェント中 1 つ。シミュレーション段階では文書ベースで OK だが、論文の提案手法を完全に reproduce するなら必要。M3 で着手予定。

### I. 「合意検出」 (優先度: 中)

§5.1 の客観指標「合意形成までに要した時間」は、議論が収束したことをどう判定するかという問題を含む。

簡易には:
- 直近 N ターンで新しい claim ノードが増えていない
- 主要立場間の embedding 距離が縮まっている
- 全員の発言が「同意」フレーズを含む

など複数の信号を組み合わせる。研究的にも面白い問題なので保留しても良い。

### J. 「外部情報の活用率」 (優先度: 中)

§5.1 客観指標。提示された情報のうち参加者が実際に発言で参照したものの割合。
embedding 類似度 (cosine > 0.7 の発話を「参照あり」とみなす) で簡易測定可能。

### K. ノード/エッジ統計の発展指標 (優先度: 低)

§5.3 の少数派生存率・極化指数・論証連鎖長など。これらは仮説の細かい検証に使うので、一気にやらず必要に応じて足していく。

### L. DocumentAgent.retrieve に embedding フィルタなし (優先度: 低)

現状全件返すだけ。data/docs/ の規模が小さいうちは問題ないが、文書数が増えると LinkingAgent への候補が膨らむ。M3 のあたりで対応で OK。

### M. 音声入力 / iPhone 連携 (優先度: 低、M5)

シミュレーション段階では不要。対面実験のための準備。

## 3. 整合性が怪しい・整理が必要な箇所

### a. ファシリテーションエージェントが Condition の info_provider に半埋まっている

現状 `ConditionFullProposal.info_provider` が「最新発話の隣接エッジを返す」を担っているが、これは本来 **ファシリテーションエージェント** の仕事。Condition は実験条件を切り替える役割なので、責務がこぼれている。

整理案:
- `FacilitationAgent` を新設し、グラフを読んで Intervention を返す
- `ConditionFullProposal` は orchestrator + facilitation を持つだけにし、info_provider は facilitation に委譲

これは M2 をある程度回した後のリファクタで OK だが、論文に書く構成と一致させる方向で動かしておきたい。

### b. 「連結エージェント」と「ファシリテーションエージェント」のコード上の境界

研究計画書 §4.1 では別物として書かれているが、現コードでは LinkingAgent (連結) のみがあり、ファシリテーションエージェントというクラスがない。
コード上に `FacilitationAgent` を作ることで、論文との対応が明示化される。

### c. L1 の「実デバイスへの通知」と「シミュレーションの prompt 注入」の混同

§4.4 L1 は「各参加者の手元デバイスに短いテキストで通知」する設計。シミュレーションでは prompt 注入として簡略化しているが、ここは将来的に「対面実装に置き換え可能なインタフェース」として整理するのが筋。
現状は `info_provider(history, persona) -> str | None` でかろうじて抽象化されている。

### d. 研究計画書 §4.4 と実装計画書 (`implementation_plan.md`) の用語ずれ

実装計画書では `presentation/` 以下に `notification.py / summary.py / retrospective.py` を置く想定だったが、これらは **未実装 (空の `__init__.py` のみ)**。L3 / L4 を実装するときにここに集約する。

### e. M2.4 構造指標 と現実装 `metrics.py` のずれ

現状の `metrics.py` は §5.1 客観指標と §5.3 発展指標の **基本部分だけ**。本来 M2.4 が完了するには、極化指数や論証連鎖長などの発展指標もここに入る予定だったが、まだ。タスク状態は「pending」のままで合っている (基本指標だけ先取りした形)。

### f. CLI と Streamlit の役割分担

CLI (`das run-session`) と Streamlit (`Live Session`, `Compare`) は同じ機能を別経路で提供している。これ自体は重複ではないが、研究的なバッチ実行 (M2.6 の多数回ラン) は CLI で叩けるようにしておきたい。現状は Streamlit からしか試せない。

## 4. 推奨される次の行動

優先順位付き。括弧内は研究計画書の対応節。

### 中期 (2〜4 週間)

5. **ファシリテーションエージェントの本実装** (§4.1) — 偏り検知・反対側補強・議論ステージ判断。これは論文の提案手法の核

### 長期 (1 ヶ月+)

## 5. 全体所感

- **M1 (システム実装)** は研究計画書の要素を 7〜8 割カバーできている。中核 (LinkingAgent + Orchestrator + 統合 AF) は機能している
- **M2 (シミュレーション評価)** は **3 条件の動作** までは到達。ただし RQ への解答にはまだ届いていない (主観指標代理計測と多数回ランが必要)
- **5 エージェント中 2 つ (Web 検索・ファシリテーション本実装) が抜けている**。Web 検索は M3 で計画通り。**ファシリテーションは半分しかない点が研究上のリスク**
- **段階A (技術検証) を経ずに段階B (シミュレーション) を回している** のは方法論的に弱い。少なくとも LinkingAgent の精度評価を 1 回挟むと安心
- **§4.4 の 4 層提示** は L1+L2 のみ。**L3/L4 の不在は論文の提案手法の不完全性として目立つ**

総じて、「コアは動く」「論文として閉じるにはあと 5 つのギャップ (ファシリテーション本実装、LLM-as-judge、多数回ラン、段階A、L3/L4) を埋める必要がある」というのが現状の正直な評価。
